ADCS漏洞汇总

发表于2023-12-29|更新于2023-12-29
|字数总计:3k|阅读时长:13分钟|阅读量:

ESC7

测试机器：HTB-Manager

前提条件：域内安装CA证书服务，域内用户权限

使用certipy来枚举证书漏洞，发现存在ESC7漏洞且Raven用户有管理CA权限。

通过Raven用户添加管理证书权限

启动SubCA证书模板，添加时默认启用

请求基于SubCA模板的证书，该请求会被拒绝，记下请求的ID即可

管理证书，发出失败的请求ID（这里的机器每10分钟证书会被重置，所以会出现失败的情况。重新添加管理证书权限即可）

重新检索已颁发的证书

KRB_AP_ERR_SKEW

可能会出现与服务器时间差距较大的情况从而无法获取证书Kerberos SessionError: KRB_AP_ERR_SKEW(Clock skew too great)，可以先关闭自动校准时间然后同步域控服务器时间即可

认证已颁发的证书

ESC1

测试机器：HTB-Escape

前提条件：域内安装CA证书服务，域内用户权限

使用certipy枚举证书漏洞

针对ESC1请求管理员证书，其中UserAuthentication为之前枚举的Template Name

根据证书获取管理员hash

KDC_ERR_PADATA_TYPE_NOSUPP

这台机器正常情况下请求TGT时会出现KDC_ERR_PADATA_TYPE_NOSUPP，表示不支持这种类型的证书认证。AD默认支持两种协议的证书认证Kerberos PKINIT和Schannel，这里就是不支持Kerberos PKINIT认证，所以可以使用Schannel认证，Schannel认证走的ldaps协议。可以使用PassTheCert来实现。

先从administrator.pfx中提取密钥和证书

获取证书所代表的用户

创建机器用户

赋予创建的机器用户DCSync权限

获取域内用户hash

ESC2 ESC3

测试机器：HTB-Escape

前提条件：域内安装CA证书服务，域内用户权限

ESC2和ESC3的漏洞机器直接在Escape这台机器上构造，在构造ESC2漏洞的时候先构造ESC3漏洞，因为ESC2漏洞环境可以使用ESC1和ESC3漏洞的利用步骤。

ESC3

利用之前ESC1拿到的administrator hash先用impacket-wmiexec登录后，修改注册表使机器能用hash登录。

之后就是注册证书模板，这里复制代理注册模板，可以用下面的路径打开

Control Pannel -> Administrative Tools -> Certification Authority -> sequel-DC-CA -> Certificate Templates(Manage) -> Enrollment Agent -> Duplicate Template

复制模板的时候需要修改几个地方，其中在Security中添加Domain Users组，并且勾选Enroll。Subject Name选择Supply in the request。General修改Template display name这里随便修改就行，我修改成ESC33。修改完成后点击OK，然后返回到Administrative Tools颁发刚才创建的模板，具体的路径为：

Control Pannel -> Administrative Tools -> Certification Authority -> sequel-DC-CA -> Certificate Templates(new) -> Certificate Template to Issue

至此环境就搭建完成，使用certipy扫描漏洞

利用ryan.cooper用户注册ESC33证书

以ryan.cooper用户身份注册到的证书模板ryan.cooper.pfx代理域管administrator注册User证书模板，来获取administrator.pfx证书

用户administrator.pfx获取管理员hash

ESC2

ESC2的漏洞环境搭建和ESC3的漏洞搭建基本相似，只不过模板换成了Workstation Authentication，并且将Extensions中的Application Policies条目清空。Security中和ESC3一样需要添加Domain Users组并且赋予注册的权限，Subject Name也是选择Supply in the request。

ESC4

测试机器：HTB-Escape

前提条件：域内安装CA证书服务，域内用户权限

ESC4的漏洞机器也直接在Escape这台机器上构造。这次只需要复制Workstation Authentication模板，并且在Security添加Domain Users有读写权限即可，其他不做修改。

枚举漏洞

ESC4因为对证书模板有写权限，所以可以覆盖模板配置使其容易受到ESC1的攻击，可以使用certipy来覆盖配置，并将原来配置保存便于恢复配置。

这时再去枚举模板漏洞就会发现ESC4模板存在1，2，3，4这几个漏洞

使用ESC1的攻击手法获取管理员证书

出现KDC_ERR_PADATA_TYPE_NOSUPP使用passthecert.py解决

恢复原来的证书配置

ESC8

测试机器：春秋云境 2022网鼎杯半决赛复盘

前提条件：域内安装CA证书服务，域内用户权限

2022网鼎杯半决赛内网环境如图下所示，18为CA服务器，13为DC服务器，35为域内机器，26为入口点机器隧道搭建在这台机器上，24和26都属于域外机器。

枚举证书模板漏洞

ESC8的利用需要用到ntlmrelayx工具监听smb请求数据，服务器环境不支持，所以需要本地启动，因为收到请求后还要重放数据到CA服务器上，就要利用端口转发和socks代理，所以在入口点使用frp搭建socks代理。而PetitPotam.py的强制认证又需要445端口，因为入口点机器获取不到root权限，所以445端口无法开启监听就需要从26这台机器上将445端口转到入口点大于1024的端口上，之后将端口转发到vps上，在vps使用ntlmrelayx开启smb监听

在入口点机器上搭建frp socks5代理，并且将本地9999端口转发到vps的9999端口上

用17010打下24这台机器后关闭smb服务，将445端口的流量转发到入口点机器的9999端口上并重启。

使用24这台机器强制认证时流量其实是到vps的9999端口上

在vps上使用代理来启动ntlmrelayx，9999端口本身不需要代理，但是relay时需要代理请求CA服务器的http://172.22.15.18/certsrv/certfnsh.asp所以需要使用proxychains开启代理，而conf中代理IP改成127.0.0.1即可

将获得的base64证书解码后添加机器用户进行RBCD获取管理员证书

Certifried（CVE-2022-26923）

测试机器：TryHackMe CVE-2022-26923

前提条件：域内安装CA证书服务，域内用户权限

首先枚举漏洞，会发现这台机器有ESC1漏洞，但本次实验不使用这个漏洞，只需要记下CA Name和DNS Name即可

添加机器账号，并且将它的DNS改成域控的DNS Name，之后申请的机器证书就为域控机器用户的证书。

如果没有修改DNS则申请的机器证书就只是当前机器用户的证书，获取不到DCSync权限

获取域控机器用户hash，进行DCSync
